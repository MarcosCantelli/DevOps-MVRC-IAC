# DEVOPS-MVRC-IAC

> Full Infrastructure as Code pipeline — automated VM provisioning, configuration, and application deployment on a private vSphere environment using Jenkins, Terraform, and Ansible.

---

## Overview

This project implements a complete DevOps pipeline for a private on-premise infrastructure. Every step — from virtual machine creation to application environment setup — is fully automated and triggered by a GitHub pull request merge into the `main` branch.

The pipeline provisions a virtual machine on VMware vSphere using Terraform, then hands it off to Ansible, which installs and configures all dependencies required to run a Java backend + Angular frontend web application served through Nginx.

---

## Architecture

```
GitHub (PR merged to main)
        │
        ▼
   Webhook (ngrok tunnel)
        │
        ▼
     Jenkins
        │
        ├── Stage 1 → Terraform   →  Provisions VM on vSphere/vCenter
        │
        ├── Stage 2 → Ansible     →  Configures VM (Java 17, Node.js 20, Nginx)
        │
        └── Post Actions          →  Reports success or destroys VM on failure
```

## Pipeline Stages

### 1. Checkout SCM
Jenkins clones the repository from the `main` branch after the GitHub webhook is received.

### 2. Checkout
Logs branch name and commit hash for traceability.

### 3. Clean Previous State
Removes any leftover Terraform state files from previous runs to ensure a clean execution.

```bash
rm -f terraform.tfstate terraform.tfstate.backup tfplan
```

### 4. Terraform Init
Initializes the Terraform working directory and downloads the vSphere provider plugin.

### 5. Terraform Plan
Generates an execution plan and saves it to `tfplan`. No changes are applied at this stage.

### 6. Terraform Apply
Applies the plan and provisions the VM on vSphere. The provider waits for the VM to obtain a network IP before completing.

### 7. Capture VM IP
Extracts the VM's IP address from Terraform outputs and stores it as a pipeline environment variable for Ansible to use.

### 8. Wait for VM to Initialize
Polls the SSH port (22) on the new VM every 10 seconds, up to 5 minutes, before handing off to Ansible. This prevents connection timeouts on fresh VMs.

### 9. Ansible — Configure VM
Runs the main playbook against the new VM. The following roles execute in order:

| Role       | What it does                                          |
|------------|-------------------------------------------------------|
| `base`     | Installs curl, wget, git, xz-utils, ca-certificates  |
| `java`     | Downloads and installs JDK 17.0.14 (Eclipse Temurin) |
| `nodejs`   | Downloads and installs Node.js 20.19.5                |
| `nginx`    | Installs Nginx and configures reverse proxy           |
| `app-user` | Creates app user, directories, validates installation |

### Post Actions
- On **success**: logs the VM IP and confirms the pipeline completed.
- On **failure**: automatically runs `terraform destroy` to clean up the VM and avoid orphaned resources.

## Nginx Configuration

Nginx is configured as both a static file server and a reverse proxy:

```
GET /          → serves Angular build from /var/www/app
GET /api/*     → proxies to Java backend on localhost:8080
```

This setup allows the frontend and backend to be served on the same port (80) without CORS issues.

## Prerequisites

### Jenkins Server
- Jenkins installed and running on port `8080`
- Plugins: `GitHub Integration Plugin`
- Tools installed: `terraform`, `ansible`, `netcat`
- SSH key pair generated at `/var/lib/jenkins/.ssh/ansible_key`

### GitHub
- Repository webhook configured pointing to `https://<ngrok-url>/github-webhook/`
- Personal Access Token (PAT) with `repo` and `admin:repo_hook` scopes

### VMware vSphere
- vCenter managing at least one ESXi host
- Ubuntu 24.04 template with:
  - VMware Tools installed and running
  - SSH enabled
  - Jenkins public key added to `~/.ssh/authorized_keys`
  - Passwordless sudo configured for the VM user

### Jenkins Credentials (configured via Manage Jenkins → Credentials)

| Credential ID    | Type        | Description                     |
|------------------|-------------|---------------------------------|
| `vsphere-user`   | Secret text | vSphere username                |
| `vsphere-password` | Secret text | vSphere password              |
| `github-pat`     | Secret text | GitHub Personal Access Token    |

## Setup Guide

### 1. Fork and clone this repository

```bash
git clone https://github.com/your-username/DEVOPS-MVRC-IAC.git
cd DEVOPS-MVRC-IAC
```

### 2. Configure `terraform/terraform.tfvars`

```hcl
vsphere_server = "your-vcenter-ip"
datacenter     = "your-datacenter-name"
cluster        = "your-cluster-name"
datastore      = "your-datastore-name"
network        = "your-network-name"
template_name  = "your-ubuntu-template-name"
vm_name        = "vm-app-devops"
num_cpus       = 2
memory_mb      = 4096
```

> `vsphere_user` and `vsphere_password` are **never** stored in this file. They are injected by Jenkins at runtime via `TF_VAR_` environment variables.

### 3. Start ngrok on the Jenkins server

```bash
ngrok http 8080
```

Copy the HTTPS forwarding URL (e.g. `https://abc123.ngrok-free.app`).

### 4. Configure GitHub Webhook

In your repository: `Settings → Webhooks → Add webhook`

```
Payload URL:   https://abc123.ngrok-free.app/github-webhook/
Content type:  application/json
Events:        Just the push event
```

### 5. Configure Jenkins credentials

Add the following credentials in `Manage Jenkins → Credentials`:

- `vsphere-user` — Secret text with your vSphere username
- `vsphere-password` — Secret text with your vSphere password
- `github-pat` — Secret text with your GitHub PAT (used in GitHub Server config)

### 6. Create the pipeline job in Jenkins

- New Item → Pipeline
- Build Triggers: `GitHub hook trigger for GITScm polling`
- Pipeline: `Pipeline script from SCM` → Git → your repo URL → branch `*/main`
- Script Path: `Jenkinsfile`

### 7. Trigger the pipeline

Push any commit to `main` (or merge a PR). Jenkins will trigger automatically.

## How the SSH Authentication Works

Jenkins generates an ED25519 key pair:

```bash
sudo -u jenkins ssh-keygen -t ed25519 -C "jenkins-ansible" \
  -f /var/lib/jenkins/.ssh/ansible_key -N ""
```

The public key (`ansible_key.pub`) is added to the VM template's `~/.ssh/authorized_keys`. Every VM cloned from that template automatically trusts the Jenkins key, allowing Ansible to connect without a password.

---

## Security Considerations

- vSphere credentials are stored in Jenkins Secrets and injected as `TF_VAR_` environment variables — they never appear in code or logs.
- The `.gitignore` excludes all Terraform state files (`*.tfstate`), plan files (`*.tfplan`), and the `.terraform/` cache directory.
- SSH access to provisioned VMs uses key-based authentication only.
- Ansible connects with a dedicated key that exists only on the Jenkins server.
- The ngrok tunnel is used for local lab purposes. In a production environment, this should be replaced with a proper firewall rule or a dedicated reverse proxy with TLS termination.
